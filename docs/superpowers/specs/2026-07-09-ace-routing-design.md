# ACE 智能路由业务经验提取系统 — 设计文档

> 日期: 2026-07-09
> 状态: 已实施

---

## 1. 背景与目标

### 1.1 当前状况

消费金融智能客服路由模块，6 个 agent 线上运行，由提示词驱动。每周标注人员随机抽样标注，产出两类数据：

- **A 类（badcase）**：`router_agent ≠ human_annotate_agent`，含 `human_note`（人工理由）+ `router_reason`（模型理由）
- **B 类（准正确）**：无人工标注，默认线上路由正确，有 `router_reason`

### 1.2 核心问题

大模型缺乏业务知识积累和经验积累，导致路由判断错误。与人类相比，模型缺失的是**判断策略**——不知道什么时候该用哪种策略、什么信号可靠什么信号是陷阱。

### 1.3 目标

从标注数据中自动提取业务经验，生成一个持续进化的 **Playbook（业务经验手册）**，注入线上路由 prompt，提升 qwen3-20b-a3b 的路由准确率。

## 3. 核心 #4：排除规则（软约束）

### 3.1 输入

`agent_info.json` 中每个 agent 的 `【注意事项】` 字段。

### 3.2 处理流程

1. DeepSeek V4 单次调用：提取所有"不属于本 agent 职责范围"的条目，每条指明正确归属
2. 人工快速审核（一次性，5 分钟）
3. 写入 Playbook `EXCLUSION THRESHOLDS` 章节

### 3.3 Playbook 形态（软约束，非硬排除）

```
## EXCLUSION THRESHOLDS

[exc-00001] ::
意图 "关闭自动续费/管理自动续费"
→ 降低 "逾期及APP使用" 优先级（阈值 +0.3），归属 "借款/会员Agent"
→ 原因: 会员业务范畴

[exc-00002] ::
意图 "提前还本期/下期账单"
→ 降低 "提前结清咨询" 优先级（阈值 +0.3），归属 "还款"
→ 原因: 提前结清仅处理一次性全部还清
```

### 3.4 在决策框架中的位置

STEP 1 —— 扫描排除规则，被命中的 agent 需要更强的正向证据才能被选中。通常能将候选从 6 个缩减到 3-4 个。

### 3.5 维护

agent_info.json 变更时，重新运行生成脚本。生成的排除规则需人工审核后替换对应章节。

## 4. 核心 #1：推理链对比学习

### 4.1 输入

A 类 badcase 中的 `human_note`（人工理由）和 `router_reason`（模型理由）。

### 4.2 核心思路

不只看"预测对不对"，而是对比**人用什么策略判断 vs 模型用什么策略判断**，提取策略层面的差异。

### 4.3 处理流程

```
A类 badcase
      │
      ▼
DeepSeek V4 批量分析（Reflector 角色）:

对每条 badcase:
  1. 分解 human_note 中隐含的判断策略（意图匹配/目的推断/排除法/上下文关联...）
  2. 分解 router_reason 中模型实际使用的策略
  3. 对比两种策略，分析为什么在这个 case 中人的策略更优
  4. 如果是策略选择错误（本应用策略A但用了策略B），提取一条"策略选择规则"

对全部 badcase:
  5. 聚类相似的策略选择规则
  6. 合并为结构化的策略条目
```

### 4.4 Reflector Prompt 关键修改

在原 ACE Reflector prompt 基础上增加：

```
**人工判断理由（教师信号）：**
{human_note}

**模型判断理由：**
{router_reason}

请对比以上两种推理路径：
1. 模型用了什么判断策略？标注者用了什么判断策略？
2. 两种策略的本质差异是什么？
3. 在这个 case 中，为什么标注者的策略更优？
4. 什么情况下应该优先用标注者的策略？
5. 如果 human_note 信息不足以判断策略（过短或仅陈述结论），标注为"信息不足"

特别注意：如果 human_note 过短（<20字），不要过度推断标注者的推理过程，
只基于原文中明确表达的信息做分析。
```

### 4.5 防护措施

- **数据量门槛**：badcase < 100 时，#1 不自动生成策略规则，只产出可读的分析报告供人工参考
- **human_note 长度检查**：过短的 human_note（<20 字）标记为"信息不足"，不做过度推断
- **所有自动生成的策略规则标记为 `helpful=0`**，等待后续 badcase 验证后才增加权重

### 4.6 Playbook 形态

```
## ROUTING STRATEGIES

[str-00042] helpful=3 harmful=0 ::
【策略选择：意图精确匹配优先】
当 query 包含可识别的操作动词/名词组合（如"更改"+"手机号"、"办理"+"停催"、
"查询"+"额度"），且该组合精确出现在某 agent 的【具体意图】列表中时，
优先使用意图关键词精确匹配策略，而非对话历史上下文关联策略。
原因：操作意图独立于对话历史，用户可能在对话中途切换话题。
来源：badcase 分析 | human_note 佐证数: 8

[str-00043] helpful=1 harmful=0 ::
【策略选择：业务目的优先】
当 query 表达了用户想达成的明确结果（借钱/还清/延期/改信息），
按最终目的匹配 agent 的核心职责，即使操作路径涉及其他 agent 的范畴。
来源：badcase 分析 | human_note 佐证数: 4
```

## 5. 核心 #2：微决策流程

### 5.1 输入

所有 human_note（来自 A 类 badcase）+ 辅助 #1 的冲突矩阵。

### 5.2 核心思路

不构建一棵覆盖全部 6 个 agent 的大决策树，而是**为每个高频混淆对构建微决策流程**。顶层路由先判断 query 落入哪个混淆对的高风险区域，再执行对应的微决策流程。

### 5.3 处理流程

```
Step 1: 从冲突矩阵获取高频混淆对（混淆次数排序，取 top-N）

Step 2: DeepSeek V4 对每个高混淆对:
  输入: 该混淆对涉及的所有 badcase 的 human_note
  任务: 从这些 human_note 中归纳出区分这两个 agent 的关键判断维度
  输出: 一个微决策流程（3-5 步）

Step 3: 组装顶层路由:
  ┌─ 执行 #4 排除规则，缩小候选
  ├─ 判断 query 最像哪个混淆对的风险区域
  ├─ 执行对应的微决策流程
  └─ 如果不在任何已知高风险区域 → 通用匹配流程

Step 4: 写入 Playbook DECISION FRAMEWORK 章节
```

### 5.4 微决策流程示例

```
### 决策分支: 还款 vs 逾期及APP使用

STEP 1: query 中是否包含精确命中的意图关键词？
  → 在"逾期及APP使用"的【具体意图】中命中 → 选逾期及APP使用（置信度 +40%）
  → 在"还款"的【具体意图】中命中 → 选还款（置信度 +40%）
  → 同时命中 → 进入 STEP 2

STEP 2: 用户的核心目的是什么？
  → 资金相关（还钱/扣款/账单）→ 还款
  → 账户操作（改信息/注销/设置）→ 逾期及APP使用
  → 不明确 → 进入 STEP 3

STEP 3: 对话历史是否提供了明确上下文？
  → 历史明确涉及还款操作且 query 中没有相反信号 → 还款
  → 历史无关或 query 表现出话题独立性 → 回到 STEP 1 的结果
```

### 5.5 防护措施

- **更新频率最低**：只在累积 ≥50 条新 badcase 时才重新生成微决策流程
- **离线验证**：更新后必须在 B 类回归集上通过反向校准（核心 #5）
- **决策树不作为硬规则**：prompt 中标注"优先按此流程判断，但如果 case 明显不符合，可以跳过"

### 5.6 关键设计决策

- 用 DeepSeek V4（离线大模型）做决策树归纳，线上 qwen3-20b 只负责执行
- 归纳 prompt 要求引用具体的 human_note 作为证据，"每步至少引用 3 条 human_note 中的具体表述"
- 如果某个 micro-decision-tree 的 human_note 佐证不足 → 标注"低置信度"，不自动上线

## 6. 核心 #3：证据质量评估

### 6.1 输入

- A 类 badcase 的 `router_reason`（模型的错误推理）
- B 类正确 case 的 `router_reason`（模型的正确推理）

### 6.2 核心思路

从 router_reason 中提取两类信号模式：

- **不可靠证据模式**（来自 badcase）：什么样的推理方式容易出错
- **可靠证据模式**（来自高质量正确 case）：什么样的推理方式正确率高

### 6.3 处理流程

```
Step 1: B类数据质量过滤
  用 DeepSeek V4 对 B 类 router_reason 做推理质量评分:
    - "推理扎实": 引用了具体的【具体意图】条目，推理链简短直接
    - "推理一般": 有一些逻辑但不严谨
    - "推理牵强": 推理链很长或多步联想，可能是蒙对的
  只有"推理扎实"的 B 类 case 纳入可靠信号分析

Step 2: 对比分析
  -- 对 badcase 的 router_reason 做"证据可靠性分析":
     分解每条 router_reason 中模型引用的证据
     判断每条证据的类型（上下文关联/职责字面匹配/意图精确匹配/...）
     标记不可靠证据类型

  -- 对高质量 B 类的 router_reason 做"证据可靠性分析":
     提取"正确推理中常见的证据类型"
     标记可靠证据类型

Step 3: 归纳信号模式
  不可靠证据模式: 来自某证据类型在 badcase 中高频出现
  可靠证据模式: 来自某证据类型在高质量正确case中高频出现
```

### 6.4 Playbook 形态

```
## EVIDENCE RELIABILITY

### 不可靠证据 — 需要谨慎对待

[evi-00001] helpful=5 harmful=0 ::
【对话历史延续信号】
模型倾向: query 中有词和历史话题相关 → 假设话题延续 → 选历史相关的 agent
风险: 用户可能切换了话题，词汇层面的关联不等于话题延续
对策: 先检查 query 是否有独立于历史的意图信号。如果有，优先用 query 自身信号。
badcase 出现次数: 12/50

[evi-00002] helpful=3 harmful=0 ::
【多步联想链】
模型倾向: A 词 → B 概念 → C agent → 选 C
风险: 每多一步联想，错误的概率就放大。联想链越长越不可靠。
对策: 优先用"一步直达"的证据（query 词 → agent 意图列表直接命中）。
badcase 出现次数: 8/50

### 可靠证据 — 正确推理的信号

[evi-00010] helpful=8 harmful=0 ::
【意图关键词精确命中】
当 query 中的操作表述直接出现在某 agent 的【具体意图】列表中，
且不是通过同义词或概念联想关联的，这是一个高可靠信号。
正确case 出现次数: 45/80（高质量正确case中的占比）
```

### 6.5 防护措施

- B 类数据必须先经过推理质量过滤，排除"蒙对的"
- 证据模式单独存放于 `EVIDENCE RELIABILITY` 章节，与策略规则（`ROUTING STRATEGIES`）区分
- 如果有新的 badcase 和某条"可靠证据模式"矛盾 → 标记该模式为"待复核"

## 7. 核心 #5：反向校准

### 7.1 输入

- A 类 badcase（验证"新规则修好了目标 case"）
- B 类准正确 case（验证"新规则不掰弯正确 case"）

### 7.2 核心思路

Curator 生成新规则 → 在 A 类上做正向验证（修好了吗？）→ 在 B 类上做反向验证（搞坏了别的吗？）→ 都通过才写入 Playbook。

### 7.3 处理流程

```
Curator 生成新规则候选

Step 1: 正向验证
  在关联的 badcase 上测试 → 修复了目标 badcase？
  → NO → 丢弃该规则
  → YES → 进入 Step 2

Step 2: 影响范围识别
  在 B 类中检索与新规则关键词相关的子集（不必全量扫描）
  使用简单关键词匹配或语义相似度过滤

Step 3: 反向验证
  对受影响子集做 forward pass
  → 有原本正确的 case 被改错？→ 规则危险！
  → 没有？→ 进入 Step 4

Step 4: 规则调整（如果 Step 3 失败）
  缩小规则范围 → 重新 Step 3
  如果无法缩小 → 标记为"高风险"，记录到 curator_operations_diff.jsonl，不自动上线

Step 5: 写入 Playbook
  helpful 初始值根据验证结果调整：
  全部通过 → helpful=1
  经过修改才通过 → helpful=0（等待后续验证）
```

### 7.4 样本量不足时的处理

通过冲突矩阵识别 B 类样本量不足的 agent 对：

- 样本量充足（≥50）：完整的反向校准
- 样本量不足（<50）：仅做规则冲突检测（检查是否与已有规则矛盾），标注"反向校准不充分"
- 样本量极低（<10）：规则标记为"低置信度"，不上线，仅作为候选供人工审核

### 7.5 防护措施

- B 类数据不是金标准（可能含漏网之鱼）。反向校准的目标是"不比现状差"，不是"100% 准确"
- 反向校准结果记录到 curator_operations_diff.jsonl，可追溯
- 定期抽取一批 case 做人工标注，逐步积累"金标准回归集"

## 8. 辅助 #1：Agent 冲突矩阵

### 8.1 做什么

统计 A+B 全部数据中每对 agent 的混淆频率，生成 6×6 混淆矩阵。

### 8.2 用途

1. 作为 #2 微决策流程的输入——决定哪些混淆对需要专门的决策分支
2. 验证训练效果——训练前后混淆矩阵的稀疏度变化
3. 指导标注采样——优先采样高混淆对的边界 case

### 8.3 实现

纯统计，一次遍历数据即可。

---

## 9. 辅助 #2：对抗样本生成

### 9.1 触发条件

仅当 badcase 总量 < 200 时启用。

### 9.2 做什么

对每个 badcase 生成 3-5 个保持相同错误模式的变体，增加训练数据多样性。

### 9.3 处理流程

```
原始 badcase → DeepSeek V4 生成变体:

生成指令:
  "保持以下路由错误模式不变，但改变 query 的表面表达方式和历史对话主题。
   确保改变了表面形式后，正确的路由目标仍然相同。"

验证步骤:
  生成后 → 用当前线上模型预测 → 预测结果是否和原始 badcase 一致（都错到同一个 agent）？
  → 是 → 保持相同错误模式 ✓
  → 否 → 丢弃该变体 ✗
```

### 9.4 标签继承

通过验证的变体，从原始 badcase 继承 `human_annotate_agent` 作为正确答案。

## 10. 远期 #1：动态案例检索 DCR

### 当前不做原因

- 需要向量数据库，改变线上架构
- 增加 prompt 长度和 token 成本
- 先验证 Playbook（压缩版经验）的效果

### 触发条件

Playbook 上线后，分析仍然判错的 case。如果发现"Playbook 规则太抽象，case 太特例"，启动 DCR。

### 大致方案

标注数据向量化存储 → 线上实时检索 top-3 相似 case → 作为 few-shot 注入 prompt。

---

## 11. 远期 #2：置信度感知路由

### 当前不做原因

- 改变线上路由架构（单级→多级）
- 大模型置信度校准是独立难题
- 偏离"从数据提取经验"的任务核心

### 大致方案

模型输出置信度 → 高置信度直接路由 → 中置信度启用 DCR → 低置信度升级到大模型或人工。

| # | 决策 | 原因 |
|---|---|---|
| 离线大模型 ≠ 线上模型 | 离线用 DeepSeek V4 做知识提取，线上 qwen3-20b 执行 | 离线要高智能，线上要低成本 |
| 排除规则是软约束 | 不直接排除，而是提升匹配阈值 | 避免机器学习到的正确规则被硬规则覆盖 |
| 混淆对微决策树 | 不一棵大树覆盖全部 | 不同混淆对的判断维度不同 |
| 决策树低频更新 | ≥50 条新 badcase 才重新生成 | 保持稳定性 |
| B 类推理质量过滤 | 只从"推理扎实"的正确 case 提取信号 | 避免学习"蒙对的"推理 |
| badcase<100 不自动生成策略规则 | #1 降级为分析报告 | 数据太少时统计不可靠 |
| badcase<200 启用对抗增强 | 解决数据稀疏 | 生成变体需通过自动验证 |
| 反向校准样本不足时降级 | 只做规则冲突检测 | 不充分的数据不做强结论 |
| 规则冲突检测优于盲目信任 | 有冲突的规则同时降权，交人工裁决 | 避免系统在不确定的方向上累积偏差 |

## 14. 成功度量

1. **离线指标**：Playbook 训练前后，在预留 badcase 测试集上的准确率提升
2. **线上指标**：Playbook 上线后，标注人员的 badcase 检出率（抽取样本中 badcase 占比）是否下降
3. **Playbook 质量**：helpful/harmful 比率，高 helpful 规则占比
4. **覆盖指标**：冲突矩阵中高频混淆对的混淆次数是否下降
